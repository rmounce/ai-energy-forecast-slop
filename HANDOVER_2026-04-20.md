## Critical Operational Notes

### Never mix checkpoints and scalers

`models/tft_price/checkpoint_best.pt` and `models/tft_price/scalers.pkl` must always be from the **same training run**. When swapping checkpoints for A/B eval, always copy the matching scalers alongside. The training script does this automatically on best-checkpoint save. See `memory/project_tft_dataset_coupling.md`.

### Never eval a new debiaser against an old TFT

Changing the OOF debiaser changes the training distribution. The old TFT trained on the old distribution — dropping a new debiaser onto it produces a train/inference mismatch and will make results worse. Any debiaser structural change requires the full pipeline: debiaser retrain → dataset rebuild → TFT retrain → eval.

### Run 011b baseline eval results are frozen

`eval/results/holistic_eval_results.csv` contains the Run 011b + binary routing results (+9.7%). Do not overwrite this file with new experiment results — save those to a named CSV instead (e.g. `holistic_eval_results_tft014_enhanced_input.csv`). Only update the main CSV when promoting a new model to production.

## Pre-Implementation Q&A

### Q1: Is `pd7_rrp` raw or debiased in training and inference?

**Raw everywhere.** The OOF debiaser (`train/train_pd_debiaser.py`) only covers PREDISPATCH (`pd_rrp`, steps 0–55). No debiasing path exists or is intended for `pd7_rrp`. The model is expected to learn PD7Day's bias characteristics directly from the `pd7_available` and `pd7_generation_hour` meta-features.

---

### Q2: Where does live inference get PD7Day data?

Use InfluxDB measurement `rp_30m.aemo_pd7day_forecast` — the same measurement that `ingest/ingest-pd7day.py` populates. Add a new helper in `forecast.py` that queries for the most recent PD7Day `run_time ≤ T` and retrieves its `rrp` values for all 144 decoder intervals. Follow the pattern of the existing PREDISPATCH InfluxDB fetch in `_execute_tft_prediction()`. Do **not** add a new NEMWeb fetch path.

---

### Q3: What timezone for `pd7_generation_hour`?

**Brisbane local time** (`Australia/Brisbane`, UTC+10 fixed, no DST). PD7Day publishes at approximately 07:20, 12:55, and 18:05 AEST. The dataset builder explicitly uses `pd7day_run_t.tz_convert("Australia/Brisbane").hour` — use the same in `_execute_tft_prediction()`.

---

### Q4: Should retro inference use "latest PD7Day ≤ forecast timestamp" or a stricter publication-time-correct lookup?

**Keep the bisect pattern** ("latest run ≤ forecast timestamp"). The PREDISPATCH parquet is already indexed by `run_time`, so bisect-right gives exactly what was published as of T. This is already publication-time-correct. The rolling eval uses the same pattern. No special tightening needed for the interim holistic eval script.

---

### Q5: What about pre-2026-02-09 windows in the holistic eval (no PD7Day)?

**Keep the full 811-window historical eval set.** The model handles `pd7_available=0` gracefully — `pd7_rrp=0`, `pd7_generation_hour=0`, `pd7_available=0`. The interim gate compares Run 014 vs Run 011b on the **same 811 windows**, so it is apples-to-apples even though ~96.5% of windows have no PD7Day. The mixed eval intentionally tests both regimes.

---

### Q6: How do rolling MPC eval baselines work — where do the forecasts come from?

- **`amber_apf_lgbm`**: reuse `eval/holistic_eval.py` dispatch logic with the baseline source. It already simulates `amber_apf_lgbm` dispatch from the existing retro forecasts and frozen actuals parquet (`eval/results/holistic_eval_actuals.parquet`).
- **Run 011b**: `eval/retro_tft_inference.py --overwrite` regenerates TFT forecasts using the active checkpoint. The frozen actuals parquet is the same file.
- **Dispatch simulation**: reuse `eval/dispatch_simulator.py` for the MPC loop. The rolling eval adds SoC carryover between steps — the one-shot eval does not carry SoC.

---

### Q7: Where is the EMHASS automation, and how do I test in shadow mode?

The day-ahead optimisation trigger is `rest_command.emhass_dayahead_optim` defined in `hass/package-emhass.yaml`. The automation that calls it lives in `hass/automation-sigenergy-emhass.yaml` (backed up in repo; live copy managed via HA).

**Shadow mode:** disable `automation-sigenergy-emhass.yaml` in HA so EMHASS optimises and publishes output entities but HA does not act on them to command the inverter. Validate by confirming `mpc_*` output entities update every 30 min without the inverter changing mode.

---

### Q8: What start date for the rolling MPC eval?

**`2026-02-09`** — first PD7Day availability in InfluxDB. This is the natural start where all three required inputs coexist: PREDISPATCH (from 2022), PD7Day (from 2026-02-09), and actuals (from 2024-03). Do not try to start earlier — there is no PD7Day data before this date, so `pd7_available` would be 0 for all windows and the eval would not exercise the new features at all.
